## Core Features

### 1. Capacity Guard
Before every registration, the backend checks how many seats are already taken. If the event is full, the registration is rejected with a clear error. No overbooking, period.

### 2. Secure Ticket Generation
Each registration produces a **UUID-based ticket** that's impossible to guess. No sequential IDs, no patterns â€” just a random, unique string tied to that specific student and event.

### 3. Role-Based Access Control
The system enforces roles at the API level:
- Students can browse and register.
- Only Organizers can create events and access the check-in portal.
- Routes are protected by authentication middleware.

### 4. Check-in Verification
At the event entrance, organizers paste a Ticket UUID into the check-in portal. The system verifies:
- Is this a real ticket?
- Is it for *this* event?
- Has this person already checked in?

### 5. Event Lifecycle
Events have a status â€” `UPCOMING`, `ONGOING`, `COMPLETED`, or `CANCELLED`. Registration is only allowed for `UPCOMING` events. Organizers can update the status as the event progresses.

## Architecture

The backend follows a **Controller â†’ Service â†’ Repository** pattern:

- **Controllers** handle incoming HTTP requests and send responses.
- **Services** contain the business logic â€” checking capacity, validating tickets, enforcing roles.
- **Repositories** talk to the database through Prisma. No raw SQL leaks into the business layer.

The codebase uses **OOP principles** in TypeScript:
- **Inheritance** â€” A base `User` class extended by `Student` and `Organizer`.
- **Encapsulation** â€” Database logic is hidden behind Repository classes.
- **Polymorphism** â€” Event behavior changes based on its status.
- **Abstraction** â€” Interfaces define contracts between layers.
